## Acceptance Criteria
- [ ] Clean, minimal welcome screen with Leblebi logo and one CTA displayed on first launch
- [ ] "Bring Your Own Key" screen with provider selection (OpenAI, Anthropic, Google, Mistral, Ollama/local, custom OpenAI-compatible endpoint)
- [ ] Provider cards with logos, expandable on tap to show API key input field
- [ ] "Test Connection" button with visual feedback (validation spinner → green checkmark or red error with human-readable message)
- [ ] Support for multiple providers with ability to set one as default (star badge indicator)
- [ ] Skip option that enables "explore mode" (UI tour, no agent execution)
- [ ] Persist first-launch state to prevent showing welcome screen on subsequent launches
- [ ] All entities generated using ZFA (Zorphy) with JSON serialization
- [ ] All Clean Architecture layers generated using ZFA generate command
- [ ] State management using ZFA-generated State classes

## Task
Implement the first-launch onboarding flow for Leblebi app.

### Screen 1: Welcome
Create a clean, minimal welcome screen that displays:
- Leblebi logo (centered, prominent)
- App tagline: "Multi-agent AI assistant. Flutter. BYOK. Open source. Runs everywhere."
- Single CTA button: "Get Started"

### Screen 2: Provider Selection ("Bring Your Own Key")
After tapping "Get Started", display the BYOK screen:

**Provider Cards:**
Display cards for each provider with their logo:
1. OpenAI
2. Anthropic
3. Google (Gemini)
4. Mistral
5. Ollama/local
6. Custom OpenAI-compatible endpoint

**Card Interaction:**
- Tap a card to expand it
- Expanded state shows:
  - Single text field for API key input (obscured by default, toggle visibility)
  - "Test Connection" button
  - Visual feedback during validation:
    - Spinner while testing
    - Green checkmark on success
    - Red error icon with human-readable error message on failure
  - "Save" button (enabled only after successful test)

**Multiple Providers:**
- User can add multiple providers
- Each saved provider appears in a list with status indicator
- One provider can be marked as default (star badge)
- Long-press or context menu to set as default, edit, or remove

**Skip Option:**
- "Skip for now" link at bottom
- Tapping skip enables "explore mode":
  - User can navigate the app and see UI tour
  - No agent execution possible
  - Banner/pill indicator showing "Explore Mode - Add API key to enable agents"

### State Management:
- Track `hasCompletedOnboarding` boolean
- Track `isInExploreMode` boolean
- Store list of configured providers with their validation status
- Store which provider is the default

### Navigation:
- After saving at least one provider → navigate to UC-1.2 (Create First Agent)
- If skipped → navigate to UC-1.3 (Quick Tour) in explore mode

## ZFA Architecture Requirements

### Entity Definition

**Provider Entity:**
```bash
zfa entity create -n Provider --field "id:String,name:String,type:String,apiKey:String?,baseUrl:String?,status:String,isDefault:bool,isValidated:bool,lastUsed:DateTime?,createdAt:DateTime,updatedAt:DateTime" --json
```

**ProviderType Enum:**
```bash
zfa entity enum -n ProviderType --value openai,anthropic,google,mistral,ollama,custom
```

**OnboardingState Entity:**
```bash
zfa entity create -n OnboardingState --field "id:String,hasCompletedOnboarding:bool,isInExploreMode:bool,defaultProviderId:String?,createdAt:DateTime" --json
```

### Clean Architecture Generation

Generate full Clean Architecture for Provider:
```bash
zfa generate Provider --methods=get,getList,create,update,delete --data --vpc --state
```

Generate OnboardingState (singleton pattern):
```bash
zfa generate OnboardingState --methods=get,update --data --vpc --state
```

### State Classes

The ZFA-generated `ProviderState` will include:
```dart
class ProviderState {
  final AppFailure? error;
  final List<Provider> providerList;
  final Provider? provider;
  final bool isGetting;
  final bool isGettingList;
  final bool isCreating;
  final bool isUpdating;
  final bool isDeleting;
  
  bool get isLoading => isGetting || isGettingList || isCreating || isUpdating || isDeleting;
  bool get hasError => error != null;
}
```

### Implementation Notes

1. **Secure Storage**: Provider API keys must be stored securely (Keychain/Keystore). Override the data source to use `flutter_secure_storage`.

2. **Validation Logic**: Implement provider connection testing in `ProviderRemoteDataSource`:
```dart
@override
Future<Provider> create(Provider provider) async {
  // Test connection before saving
  final isValid = await _testConnection(provider.apiKey, provider.baseUrl);
  if (!isValid) {
    throw ValidationFailure('Invalid API key');
  }
  // Store securely
  await _secureStorage.write(key: provider.id, value: provider.apiKey);
  return provider.copyWith(isValidated: true);
}
```

3. **Singleton OnboardingState**: Use a single document/key for onboarding state.

4. **DI Registration**: Register repositories in your DI container:
```dart
// Using get_it or similar
getIt.registerLazySingleton<ProviderRepository>(
  () => DataProviderRepository(getIt())
);
```
